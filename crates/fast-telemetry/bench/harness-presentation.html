The Problem

Six architectural bottlenecks

The common pattern is per-call shared mutable state. Under a multi-threaded harness, that turns observability into the bottleneck.

Metrics
Runtime labels
OTel hashes and normalizes attribute sets at record time. fast-telemetry resolves known labels to fixed slots.
Metrics + Logging
Shared contention
Central maps and writers serialize worker threads. Thread-local and sharded paths absorb writes locally.
Metrics
Dynamic dispatch
Trait-object instrument chains add vtable hops. Concrete counters inline to a small hot path.
Logging
Heap pressure
Formatted strings and runtime filters add work before the sink boundary. Typed fields defer formatting.
Spans
ID + refcounts
Per-span ID generation and shared ownership add synchronization. Local PRNG and ownership remove that cost.
Build
Runtime gating
Disabled instrumentation should disappear. Compile-time macros can remove it from generated code.

Hot path per record

1
Select shard
thread_id & (N - 1)
One bitmask operation chooses the current thread's cell.
2
Isolate cache line
CachePadded<AtomicIsize>
Each hot counter cell is separated from neighboring writes.
3
Record value
fetch_add(value, Relaxed)
No lock acquisition and no cross-thread map lookup on the request path.
4
Merge later
sum(shards)
Export pays the aggregation cost outside the recording path.
Interpretation: the measured gap comes from moving synchronization and aggregation out of the per-request write path.

Reproducibility

How to rerun the proof

The deck points at committed result files. To refresh the numbers, rerun the harness and regenerate this slide data from the latest summaries.

Full metric matrix
$ ./bench/run-bench-suite.sh --preset full --threads 10 --runs 1
Robust span sweep
$ ./bench/run-span-bench.sh --scenario all --threads 16 --runs 5
Focused counter rerun
$ ./bench/run-cache-bench.sh --entity counter --profile hotspot --threads 16 --runs 5 --modes fast,metrics,otel
